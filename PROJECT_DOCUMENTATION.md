## 1. Project Overview

**Corporate Management** is a full-featured **Human Resource Management System (HRMS)** built as a Single Page Application (SPA) using Angular 21. It is the frontend half of a full-stack system — the backend is a **.NET Core Web API** using **Dapper ORM** and **SQL Server Stored Procedures**.

The portal serves as a company's internal HR platform where employees, managers, and HR staff each get their own role-specific dashboard and tools.

### Core Modules

| Module | Description |
|---|---|
| Authentication | JWT-based login, signup, role-based access |
| Employee Self-Service | Dashboard, leave, attendance, timesheet, profile |
| Manager Tools | Team management, approvals, performance |
| HR Operations | Payroll, recruitment, leave & attendance management |
| Notifications | Real-time in-app notification panel |
| Recruitment | Job postings, candidate pipeline, resume management |

## 5. User Roles

The system has 4 roles stored as `roleId` in the database and embedded in the JWT token:

| Role ID | Role Name | Landing Page After Login |
|---|---|---|
| 1 | Admin | No dedicated dashboard (redirects to 404) |
| 2 | Employee | `/dashboard/dashboardpage` |
| 3 | Manager | `/dashboard/managerdashboard` |
| 4 | HR | `/dashboard/hrdashboard` |

Roles are extracted from the JWT token using `jwtDecode()`. The claim key checked is:
- `role` (standard)
- `http://schemas.microsoft.com/ws/2008/06/identity/claims/role` (Microsoft .NET claim)

## 7. Route Guards

### authGuard (`guards/auth-guard.ts`)

The simplest guard. Checks if a JWT token exists in `localStorage`.

```typescript
const token = localStorage.getItem('token');
if (token) return true;
else { router.navigate(['/login']); return false; }
```

- Applied to: the `/dashboard` parent route and all child routes
- Effect: unauthenticated users are always redirected to `/login`

### guestGuard (`guards/guest-guard/guest-guard.ts`)

The inverse of authGuard. Prevents already-logged-in users from visiting public pages.

```typescript
if (auth.isLoggedIn()) { router.navigate(['/dashboard']); return false; }
return true;
```

- Applied to: `/login`, `/signup`
- Effect: logged-in users are redirected to `/dashboard`

### roleGuard (`guards/role-guard/roleguard-guard.ts`)

The most sophisticated guard. Reads the user's role from the JWT and compares it against the route's `data.roles` array.

```typescript
const expectedRoles = route.data?.['roles'] as string[];
const userRole = auth.getRole();

if (!expectedRoles.includes(userRole)) {
  // redirect to the user's own dashboard
  if (userRole === 'Employee') router.navigate(['dashboard/dashboardpage']);
  else if (userRole === 'Manager') router.navigate(['/dashboard/managerdashboard']);
  else if (userRole === 'HR') router.navigate(['/dashboard/hrdashboard']);
  return false;
}
return true;
```

- Applied to: every child route under `/dashboard`
- Effect: wrong-role users are silently redirected to their own dashboard (no error shown)

## 8. Authentication Flow

### Login

```
1. User fills email + password on /login
2. UserService.login(data) → POST /api/User/Login
3. Backend returns { token: "JWT...", userId: 42 }
4. Authservice.setToken(token) → localStorage.setItem('token', token)
5. Authservice.setUserId(userId) → localStorage.setItem('userId', userId)
6. LeaveService.InitillizeLeaveBalanceApi() → ensures leave balances exist for user
7. jwtDecode(token) → extract role
8. Navigate to role-specific dashboard
```

### Logout

```
1. Authservice.Logout()
   → localStorage.removeItem('token')
   → localStorage.removeItem('role')
   → localStorage.removeItem('userId')
   → _loggedIn signal set to false
2. UserService.logout(userId) → POST /api/User/Logout
3. Navigate to /login
```

### Authservice (`services/Auth-service/authservice.ts`)

| Method | Description |
|---|---|
| `isLoggedIn()` | Returns signal value (true/false) |
| `getToken()` | Returns JWT from localStorage |
| `setToken(token)` | Stores JWT, sets loggedIn signal to true |
| `setUserId(id)` | Stores userId in localStorage |
| `getRole()` | Decodes JWT, returns role string |
| `getEmail()` | Decodes JWT, returns email |
| `getUserId()` | Returns userId from localStorage |
| `Logout()` | Clears all auth data from localStorage |

## 10. State Management

The project uses **Angular Signals** — Angular's built-in reactive primitives (no NgRx, no Redux, no BehaviorSubject patterns).

### Signal Patterns Used

```typescript
// Declare mutable state
myLeaves = signal<any[]>([]);
isLoading = signal(false);
userId = signal<any>(null);

// Derived/computed state (auto-updates when dependencies change)
pendingLeaves = computed(() =>
  this.myLeaves().filter(l => l.status?.toLowerCase() === 'pending').length
);

// Update state
this.myLeaves.set(apiResponse);           // replace
this.myLeaves.update(list => [...list]);  // transform
```

### Persistence

- JWT token → `localStorage.getItem('token')`
- User ID → `localStorage.getItem('userId')`
- Recruitment stage overrides → `localStorage.getItem('hrms_recruit_stages')`

### No Global Store

Each component manages its own local state. There is no shared store between components — data is re-fetched on each page load.

## 18. Leave Management Workflow

### Full Approval Flow

```
Employee applies for leave
  → status: "pending"
  → Notification sent to assigned manager

Manager reviews
  → Approve → status: "managerapproved" → Notification to HR
  → Reject  → status: "managerrejected" → Notification to Employee

HR reviews (only sees "managerapproved" leaves)
  → Approve → status: "approved" → Notification to Employee
  → Reject  → status: "rejected" → Notification to Employee

Leave balance deducted automatically on final approval
```

### Leave Types & Default Balances

| Leave Type | Default Balance | Notes |
|---|---|---|
| Annual Leave | 12 days | Per year |
| Sick Leave | 8 days | Per year |
| Emergency Leave | 5 days | Per year |
| Comp Off | 5 days | Per year |
| Unpaid Leave | NULL (unlimited) | No balance tracking |

### Leave Balance Rules

- Each user gets their own balance record (per leave type)
- Balance is initialized on first login via `initializeUsersLeaveBalance` API
- Only Employee, Manager, HR get leave balances (not Admin)
- Validation: cannot apply for more days than remaining balance
- Exception: Unpaid Leave has no limit

### Leave Deduction in Payroll

Unpaid leave days are deducted from salary at payroll generation time:

```
Per Day Salary = Basic Salary ÷ Total Days in Month
Leave Deduction = Per Day Salary × Unpaid Leave Days
```

## 19. Payroll Calculation Logic

### Salary Components

```
Gross Salary = Basic Salary + Allowances (HRA + Other Allowance)

Total Deductions = PF + Tax Deduction + Leave Deduction

Net Salary = Gross Salary − Total Deductions
```

### Leave Deduction Formula

```
Per Day Salary = Basic Salary ÷ Total Days in Month
Leave Deduction = Per Day Salary × Unpaid Leave Days
```

Unpaid leave days are fetched from:
```sql
SELECT SUM(TotalDays) AS unpaidleaves
FROM LeaveRequests
WHERE UserId = @userId AND RequestType = 'Unpaid Leave'
```

### Payroll Generation Window

```
Rule: HR can generate payroll for month M only during days 1–10 of month M+1.

Example:
  March payroll → can generate April 1 to April 10
  After April 10 → March payroll is LOCKED

States:
  isPayrollLocked() === false    → window open, can generate
  isPayrollLocked() === true     → window closed, locked
  isPayrollLocked() === 'future' → month not yet finished
```

### Payroll Status Values

| Status | Description |
|---|---|
| `Ready` | Salary structure exists, payroll not yet generated |
| `Generated` | Payroll created, awaiting payment |
| `Paid` | Marked as paid |
| `Locked` | Generation window has closed |
| `No Salary` | No salary structure configured for this employee |

## 24. Known Gaps & Improvement Areas

### Missing Features

| Issue | Details |
|---|---|
| No HTTP Interceptor | JWT token is never automatically attached as `Authorization: Bearer` header. Each service makes raw HTTP calls. The backend likely validates via another mechanism. |
| No production environment | `environment.prod.ts` does not exist. Production builds will use the localhost API URL. |
| Empty test files | All `*.spec.ts` files exist but contain no test implementations. |
| Admin role incomplete | Admin has a `roleId` of 1 and is handled in guards, but there is no admin dashboard or admin-specific routes. Login redirects admin to a non-existent route (`/fgfhtjgf`). |
| No error interceptor | 401/403 responses are not globally handled. Each component handles errors individually. |
| No request caching | Every page load re-fetches all data from the API. |

### Security Considerations

| Issue | Recommendation |
|---|---|
| Token in `localStorage` | Vulnerable to XSS attacks. Use `httpOnly` cookies instead. |
| No CSRF protection | Add CSRF tokens for state-changing requests. |
| Client-side role validation | Role guard can be bypassed. Backend must enforce role checks on every endpoint. |
| Hardcoded localhost URL | `https://localhost:44346` is hardcoded in `AttendanceService.exportAttendanceReport()` — should use `environment.ApiUrl`. |

### Code Quality Notes

| Item | Details |
|---|---|
| `sample.md` | Contains raw development prompts and notes — not documentation. Should be removed or moved to a private location. |
| `sql_recruitment.md` | Contains brief dev notes. Not useful as documentation. |
| `any` types | Many service methods use `any` type instead of typed interfaces. Adding proper TypeScript interfaces would improve maintainability. |
| No HTTP interceptor for auth | Adding one would remove the need to manually handle auth in each service. |
| `console.log` statements | Several `console.log` calls remain in production code (e.g., `dashboard.ts`). Should be removed. |

### Potential Enhancements

- Add virtual scrolling for large employee/leave lists
- Implement request caching with `shareReplay` or a cache service
- Add service worker for offline support
- Add comprehensive error handling with user-friendly messages
- Implement real-time notifications via WebSocket or SignalR

- Complete Admin role dashboard
- Add unit and integration tests
